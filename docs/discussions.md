# 讨论：Baseline vs Multi-Agent — 视觉质量的反直觉差距及其启示

> 本文基于 baseline（deepseek-v4-flash, 单次直出）与 multi-agent（deepseek-v4-pro, 4 Agent + 知识检索 + 渲染验证 + 反馈闭环）的实际 SVG 输出对比，分析为何更简单的系统在视觉美感上反而显著优于更复杂的系统，并提出在不牺牲量化评估能力前提下的优化方案。

---

## 1. 现象描述

全部 5 个测试样本的 baseline 与 multi-agent 输出放置在一起进行肉眼对比时，一个反直觉的结论浮现：

> **Baseline（单次 LLM 调用，无任何多 Agent 架构）生成的 SVG 在视觉美观度、配色灵性、排版精致度、图标选用、整体感染力上，明显优于 multi-agent 系统的最终输出。**

这一差距在 sample4（咖啡生产链）和 sample1（LLM 原理）上尤为突出。然而 multi-agent 系统在内容准确性（知识检索）、事实可靠性（fallback DB 验证）、结构完整性（NER/RE 结构化输出）、数值精确性（柱状图比例尺）、以及可追溯性（IR 日志）上明显胜出。

下面的讨论试图解释**为什么**会出现这种分化，以及**如何**在保留 multi-agent 量化评估优势的前提下缩小视觉质量差距。

## 3. 根因分析：为什么更复杂的系统反而更不"好看"？

### 3.1 「设计委员会」效应（Design by Committee）

多 Agent 流水线将"设计一张图"这一整体性创意任务**拆解为 4 个独立优化子任务**：

```
Agent 1: 优化内容抽取准确性     → Content IR（语义正确性）
Agent 2: 优化布局规范性         → Layout IR（空间正确性）
Agent 3: 优化 SVG 语法正确性     → SVG（受 IR 约束填充）
Agent 4: 优化 6 维评分表现      → 修改建议（指标正确性）
```

**没有任何 Agent 被赋予"让这张图好看"的整体责任。** 每个 Agent 在各自维度上做到 satisficing（"够好就行"），最终产物是一个各维度评分不低但视觉上缺乏整体协调性和灵魂的拼凑品。

当一个人（或一个模型）独自完成从内容策划到 SVG 编码的全过程时，它可以在"放多少技术细节"和"留多少视觉呼吸空间"之间做整体权衡。当 4 个 Agent 各自负责一个环节时，这种权衡不复存在——Agent 2 不知道 Agent 3 最终会把某个文本渲染成多宽，Agent 3 不知道 Agent 1 抽取的某个实体在原始 prompt 中有多重要的情感分量。

### 3.2 IR 中间表示对创造性表达的隐性压制

Content IR 和 Layout IR 是为**可控性和正确性**设计的——它们捕捉结构化信息（实体类型、关系方向、百分比坐标、元素色彩），但**无法捕捉**"这张图应该给人一种温暖/科技/学术/童趣的感觉"这类整体美学意图。

Baseline 自发加入的以下高质量视觉元素，**不出现在任何 IR Schema 或 Agent 指令中**：

- Emoji 图标（🌱🔥☕🧠⚙️🚀）——赋予视觉个性和情感温度
- 趣味事实细节（"每棵树每年仅产约 0.5kg 咖啡豆"）——提升内容附加值
- 诗性页脚文字——创造情感共鸣和记忆点
- 卡片左侧色条——增加视觉节奏和层次
- 装饰性背景圆——增强空间深度感
- 情境化配色（咖啡链的暖调大地色随时间轴逐渐变深）——颜色成为叙事工具

这些不是被任何规则明确"禁止"的——但它们落入了过度结构化的工作流所创造的设计真空。当 Agent 3 被要求"按照 Layout IR 的 x_pct/y_pct/color_scheme 生成 SVG"时，它自然地将注意力集中在**忠实执行规范**上，而非**主动装饰和美化**。

### 3.3 Thinking Mode 对视觉创意的意外惩罚

Multi-agent 系统对 Agent 3 启用了 DeepSeek 原生 thinking mode + reasoning_effort=high：

```
Agent 3 (thinking enabled): ~150-190s, 大量 token 消耗在:
  "Layout IR 要求 section_1 位于 x=5%, y=10%, w=90%, h=25%..."
  "元素 text_block_3 的 size_hint 是 min_width=200..."
  "我需要确保所有坐标在 viewBox 范围内..."
  → 分析性思维模式 → 保守的、规范的、但缺乏灵性的 SVG

Baseline (thinking disabled): ~20s, 全部 16384 token 直接用于 SVG 输出:
  "让我设计一张关于咖啡的温暖的信息图..."
  → 创造性思维模式 → 丰富的、有个性的、令人愉悦的 SVG
```

Thinking mode 增强**分析推理**但可能削弱**创造性发散**——它对需要严谨推理的任务（数值计算、逻辑推导）有效，但对视觉设计这类需要直觉和审美判断的任务，过度分析反而导致保守和呆板的输出。

### 3.4 Agent 间信息传递的逐级衰减

```
用户 Prompt（完整的语义空间 + 隐含的情感基调 + 风格倾向）
    ↓ 有损压缩
Content IR（仅保留: intent + entities + relations + key_points）
    ↓ 有损压缩
Layout IR（仅保留: chart_type + canvas + color_scheme + sections + elements）
    ↓ 忠实还原
SVG（基于 Layout IR 的约束，填充具体的 XML 代码）
```

每一步信息传递都是**有损的**。原始 prompt 中的风格倾向（"通俗易懂"→ 应该用轻松的语气和可爱的图标）、情感基调（"发展历程"→ 应该用庄重的配色和时间感）、叙述节奏——这些在 Content IR 中存在（如 `target_audience: general`），但在后续的 Layout IR → SVG 转换中逐渐被稀释为抽象的设计规范（"学术稳重配色"、"正文 14-16px"）。

Baseline 绕过了全部压缩步骤——原始 prompt 的所有隐含创意潜力，包括 LLM 从 prompt 措辞中推断出的风格、语气、节奏，全部直达 SVG 生成。

### 3.5 标准化 vs 情境化的配色策略

| 系统 | 配色机制 | 示例 |
|------|----------|------|
| **Multi-Agent** | Agent 2 从 4 套预定义配色方案中**机械选择**一个（科技蓝紫 / 自然清新绿 / 学术紫 / 数据多色） | 咖啡链 → "自然清新感"绿蓝橙；数据对比 → "数据可视化"蓝紫绿 |
| **Baseline** | 模型**自主为每个样本创造情境化配色** | 咖啡链 → 暖调大地色（米黄→棕→橄榄绿），颜色讲述咖啡豆从生到烘的旅程；数据对比 → 平台品牌色（YouTube 红、TikTok 青粉、Kuaishou 橙） |

预定义的"正确"配色在色轮上是和谐的，但它们无法匹配模型自由发挥时产生的**情境化色彩叙事**——baseline 的咖啡链配色之所以令人愉悦，不是因为单一颜色的选择，而是因为 5 个步骤的颜色（绿→棕→深棕→银灰→金）形成了一个微妙的视觉时间线，这与咖啡生产链的物理变化过程（绿生豆→浅烘→深烘→研磨粉→金色咖啡液）暗合。

### 3.6 Satisficing 天花板：当"足够好"成为上限

Multi-agent 流水线的设计隐含了一个**通过线逻辑**：

```
Agent 4: score ≥ 8.0 → pass → 停止精炼
```

这个阈值是质量保证的安全网（防止坏输出），但它同时也是**质量天花板**（阻止好输出变成更好的输出）。Agent 3 在生成 SVG 时知道 Agent 4 会审核——这会产生一种隐式的"求稳"倾向：**不要冒险做太大的设计创新，因为 Agent 4 可能会因为某个创新导致的次要布局问题而判不通过。**

Baseline 没有审核者、没有通过线、没有"够好就行"——模型在完全自由的空间中追求它所理解的"最好"，不受"会不会被下游 Agent 拒绝"的预期约束。

### 3.7 架构复杂度的隐性代价（Meta-Level）

最反直觉的发现：

```
deepseek-v4-flash + 无 thinking + 精简 prompt  → 更美的 SVG
deepseek-v4-pro  + thinking   + 多 Agent 协同  → 更正确的 SVG
```

**更强的模型 + 更强的推理 + 更复杂的架构并没有产生更好的视觉输出。** 这说明对于视觉创意生成这类"整体性任务"，架构复杂度本身可能成为负担——分解、标准化、评估的每一步都在系统性地压制创意自由度。这不是任何一个 Agent 的"错误"，而是**架构设计本身的取舍**：用创意换可控性、用灵性换可审查性。

## 4. 优化方案：在不放弃量化评估的前提下提升视觉质量

以下优化方案遵循核心原则：**保留 multi-agent 架构的量化评估能力（Agent 1 NER/RE、Agent 4 六维评分、渲染验证），同时唤醒被过度结构化所压制的创造性**。

### 4.1 方案 A：Agent 2 去模板化 — 自主设计评估配色与风格

**当前问题**：Agent 2 从 4 套预定义配色方案中机械选择一个，缺乏对具体内容的语境化适应。

**优化方向**：移除固定配色模板，让 Agent 2 自主设计并**论证**其配色和视觉风格选择。

**具体修改**（`agent2_system.txt`）：

```
旧（模板化）：
  从以下 4 套配色中选择最合适的：
  1. 现代科技感（蓝紫）→ 适合技术概念
  2. 自然清新感（绿蓝橙）→ 适合流程科普
  3. 学术稳重感（紫深蓝红）→ 适合历史学术
  4. 数据可视化（五色板）→ 适合数据对比

新（自主设计 + 论证）：
  为以下每个视觉要素自主设计配色方案，并在 rationale 字段中解释你的选择理由：
  
  必须独立决策的要素（不可照搬模板）：
  - 主色调（primary）：基于内容的情感基调选择（温暖/冷静/活力/庄重/趣味）
  - 辅助色（secondary）：与主色调形成互补或类比关系
  - 强调色（accent）：用于关键数据点或核心概念的视觉突出
  - 背景策略：纯白 vs 浅色渐变 vs 质感纹理（选择并说明理由）
  - 渐变策略：标题栏是否使用渐变？节点是否使用渐变？（选择并说明理由）
  
  不需要独立设计的（按标准规范）：
  - text_primary / text_secondary：由背景色自动推导（WCAG 对比度 ≥ 4.5:1）
  - 排版层级：6 级字号体系不变
  
  在 color_scheme.rationale 字段中，用 2-3 句话解释：
  1. 为什么选择这个主色调？（与内容的情感关联是什么？）
  2. 配色如何辅助信息层次？（如何用颜色区分不同类型的信息？）
  3. 如果使用渐变，渐变方向的选择理由是什么？
```

**预期效果**：Agent 2 从"配色选择者"变为"配色设计师"，每种颜色都有内容理由而非模板匹配。这会传递到 Agent 3 的生成中，因为色板不再是通用模板而是定制设计。

**不牺牲的量化评估**：Agent 4 仍然评估 aesthetics 维度（score 0-10），只是评估标准从"是否符合模板"变为"配色是否有内容理由、视觉层次是否清晰"。

### 4.2 方案 B：Agent 3 创意权重提升 — 从"执行者"到"共同创作者"

**当前问题**：Agent 3 被定位为 Layout IR 的"忠实执行者"，虽然文档说"有最终调整权"，但实际上 prompt 中大量的"必须按 IR 执行"约束压制了创意主动性。

**优化方向**：重新定义 Agent 3 的角色——从"IR 翻译器"变为"以 IR 为参考框架的视觉设计师"。

**具体修改**（`agent3_system.txt` + `svg_coder.py`）：

```
旧（执行者角色）：
  ## 布局规范（来自 Layout Planner）
  - 图表类型: {chart_type}
  - 画布: {width} × {height}
  
  ### 配色方案: {color_scheme.name}
  - 主色: {primary}, 辅色: {secondary}, 强调色: {accent}
  
  ### 空间分区 (N 个区域):
  - section_1 (type): x=X% y=Y% w=W% h=H%
  ...

新（共同创作者角色）：
  ## 设计框架（来自 Layout Planner — 作为参考，非硬性约束）
  
  Layout Planner 提供了以下设计建议。你可以而且应该：
  1. 在保持整体结构的前提下，主动为每个区域添加装饰性元素
  2. 为每个步骤/概念选择合适的 emoji 或简笔图标（几何图形组合）
  3. 在卡片内添加趣味性细节或科普小知识（2-3 条/卡片）
  4. 在底部添加一句总结性/情感性的话语
  5. 如果 Layout IR 的某个位置会导致元素拥挤，主动增加间距或调整尺寸
  
  你必须主动添加（不在 Layout IR 中指定，但你自己判断并添加）：
  - [ ] 至少 5 个装饰性元素（emoji / 几何图标 / 背景纹理圆 / 分隔线）
  - [ ] 每个内容区域至少 1 条趣味细节或补充说明
  - [ ] 底部总结或点睛语句
  - [ ] 如果图表类型适合（流程图/时间线），为不同步骤使用区别化的色彩而非统一色
```

**预期效果**：Agent 3 从被动执行变为主动设计。必须添加的 checklist 确保最低装饰/趣味密度，缩小与 baseline 的"信息附加值"差距。

**不牺牲的量化评估**：Agent 4 的 information_completeness 维度已包含对信息充分性的评估；aesthetics 维度已包含对视觉层次的评估。渲染验证确保新增的装饰元素不引入 XML/坐标/对比度错误。

### 4.3 方案 C：双轨生成 + Agent 4 选优

**当前问题**：单轨道流水线，Agent 3 只生成一个版本，Agent 4 只能"通过/修改"——无法在多个不同的设计方案中选最优。

**优化方向**：引入双轨（或三轨）并行生成机制——一个轨道用现有 IR 约束（保证正确性），另一个轨道放宽约束（追求创意），由 Agent 4 在两个方案中选优。

**具体设计**：

```
                     ┌── 轨道 A: 标准版（IR 强约束，Agent 3 按 Layout IR 执行）
Agent 1 → Agent 2 ──┤
                     └── 轨道 B: 自由版（IR 弱约束，Agent 3 仅参考 Content IR + 原始 prompt，自主规划布局和配色）
                         │
                         ▼
                    两条轨道的 SVG 都经过渲染验证
                         │
                         ▼
                    Agent 4: 对两条轨道的 SVG 分别评分
                         │
                         ├── 选择 aesthetics 得分更高的版本
                         └── 如果自由版的内容准确性 < 7/10，回退到标准版
```

**实现要点**：
- 轨道 B 的 Agent 3 system prompt 接近 baseline 风格——精简、自由、鼓励创意
- 两个轨道并行运行，不增加等待时间（轨道 B 已在 Agent 1/2 运行期间并行执行）
- Agent 4 对比评审：比较两版的 aesthetics 和 content_accuracy，做加权决策

**预期效果**：当自由版在内容准确性上没有严重退步时（content_accuracy ≥ 7），选择视觉效果更好的版本。这保留了 Agent 1/2 的知识检索和内容分析价值（轨道 B 仍受益于 knowledge_supplement），同时释放了 Agent 3 的创意潜力。

**不牺牲的量化评估**：两条轨道都经过渲染验证 + Agent 4 六维评分，选优决策基于量化指标而非主观判断。

---

### 4.4 方案 D：Agent 4 补充"创意密度"评估维度

**当前问题**：Agent 4 的 6 个评估维度全部是"防守型"的（检查是否有错误、是否有遗漏），缺少对"进攻性"创意质量的评估。

**优化方向**：在 Agent 4 的审查维度中新增第 7 维——**创意密度**（creativity_density），不改变其他 6 维的评分逻辑。

**具体修改**（`agent4_system.txt`）：

```
新增第 7 审查维度：创意密度（creativity_density）

评估 SVG 中超出"功能性需求"的创意表达丰富度：

评分标准（0-10）：
  0-2: 纯功能性——仅有标题 + 文字 + 基础几何形状，无任何装饰、图标、细节
  3-4: 基本装饰——有少量装饰元素，但单调重复
  5-6: 中等创意——有 emoji/图标，配色有区分度，有底部总结
  7-8: 丰富创意——有情境化配色、个性化图标、趣味细节、叙事性视觉元素
  9-10: 卓越创意——配色讲述故事、图标形成视觉语言系统、细节令人惊喜、
         底部有情感化点睛之笔、整体有"设计师作品"而非"模板生成"的感觉

具体检查项：
  - [ ] SVG 中使用了多少个装饰性元素？（emoji / 几何图标 / 纹理 / 分隔线）
  - [ ] 配色是否有超出"套用模板"的情境化考量？
  - [ ] 是否有趣味性细节或科普小知识？
  - [ ] 是否有底部总结/情感化语句？
  - [ ] 文字排版是否有节奏变化（如关键数字用更大字号、引用用斜体）？

注意：创意密度 ≠ 元素堆砌。装饰适度、有品位、与内容相关是加分项；
过度装饰导致视觉混乱是减分项。

在 needs_regeneration 判断中：
  - creativity_density < 4 且 pass=true → 建议性修改（不阻塞通过，但记录在 suggestions 中）
  - creativity_density < 3 且其他维度全部通过 → 视为 needs_regeneration=true，
    regeneration_focus 中包含 "visual_enrichment"
```

**预期效果**：Agent 4 从"只找问题"变为"也找创意不足"，给后续的 Agent 3 再生提供明确的"添加视觉丰富度"指令。这在不降低其他维度门槛的前提下，提升了系统对视觉质量的敏感度。

**不牺牲的量化评估**：新增维度是**上行推动**（推动质量更好），而非**下行放宽**（降低通过门槛）。其他 6 维的评分标准和 pass 阈值完全不变。

---

### 4.5 方案 E：合并 Agent 2+3 — 合并布局规划与 SVG 生成

**当前问题**：Agent 2（布局规划器）和 Agent 3（SVG 生成器）的职责分离是视觉质量损失的主要来源——Agent 2 在不知道具体文本渲染尺寸的情况下做布局决策，Agent 3 在 Layout IR 的强约束下做 SVG 编码。两者之间的信息不对称导致了大量保守的、模板化的设计。

**优化方向**：将 Agent 2 和 Agent 3 合并为一个 Agent（Layout-to-SVG 统一生成器），**在一个 LLM 调用中同时完成布局规划和 SVG 编码**。保留 Agent 1（内容分析 + 知识检索）和 Agent 4（质量审核）作为流水线的首尾。

**具体设计**：

```
修改前: Agent 1 → KB → Agent 2 → Agent 3 → Validator → Agent 4
修改后: Agent 1 → KB → Agent 2+3 (合并) → Validator → Agent 4
```

**Agent 2+3 的 system prompt 设计要点**：
- 输入：Content IR（含 knowledge_supplement）+ 原始用户 prompt + 设计规范参考
- 输出：直接生成 SVG XML（不再经过 Layout IR 中转）
- 角色定义："你是一名信息图设计师。根据内容分析结果，直接设计并编码完整的 SVG 信息图。你同时负责空间布局、配色选择、图标设计、和 SVG 编码。"
- 关键自由：不需要遵循任何 Layout IR 约束（因为 Layout IR 不再存在），模型在单次调用中从内容到视觉端到端生成
- 保留的结构：仍然读取 Content IR 中的 entities、relations、chart_type 建议、knowledge_supplement——知识检索的价值被保留

**预期效果**：消除 Agent 2→3 的信息损失，模型在一个上下文中同时处理"怎么布局"和"怎么画"，创意决策更连贯。同时减少了 1 次 API 调用（Agent 2 被合并），总耗时下降。

**风险**：Agent 2+3 的单次调用需要处理更复杂的任务（同时规划布局和编码 SVG），可能对模型能力要求更高。需要增加 max_tokens 到 32768。

**不牺牲的量化评估**：Agent 1 的 NER/RE 结构化输出 + knowledge_supplement 仍然存在且被传递。Agent 4 的 6 维评分 + 渲染验证仍然在 SVG 生成后进行质量把关。反馈闭环仍然是 Agent 4 → Agent 2+3。

---

### 4.6 方案优先级与组合建议

| 优先级 | 方案 | 实施成本 | 预期视觉提升 | 评估能力影响 | 风险 |
|:---:|------|:---:|:---:|:---:|------|
| 🔴 P0 | **A: Agent 2 去模板化** | 低（仅修改 prompt） | 中高 | 无 | 低 |
| 🔴 P0 | **B: Agent 3 创意权重提升** | 低（修改 prompt + 代码） | 高 | 无 | 低 |
| 🟡 P1 | **D: Agent 4 创意密度维度** | 低（仅修改 prompt） | 中 | 增强 | 低 |
| 🟡 P1 | **C: 双轨生成 + 选优** | 中（新增并行轨代码） | 中高 | 增强 | 中（成本翻倍） |
| 🟢 P2 | **E: 合并 Agent 2+3** | 高（架构重构） | 高 | 无 | 中高（需重新调试 prompt） |

**推荐路径**：

1. **立即实施**（P0）：A + B + D —— 全部是 prompt-level 改动，不需要修改流水线代码逻辑。去模板化 + 创意权重提升从两个方向（上游 Agent 2 的配色设计自由 + 下游 Agent 3 的装饰创意自由）同时放大创造性空间。创意密度维度为 Agent 4 提供"推动更好"的正向评估能力。

2. **验证后实施**（P1）：C —— 在 P0 改动验证有效后，加入双轨选优机制。如果 P0 结果已经接近 baseline 的视觉质量，C 可以作为额外保险（始终保留一个安全的标准版本）。

3. **长期考虑**（P2）：E —— 如果 P0+C 的组合仍无法完全弥合视觉差距，说明 Agent 2↔3 分离本身就是最大的创意瓶颈，此时应实施合并方案。这需要重构 orchestrator.py（移除 Agent 2 调用、新增 Agent 2+3 合并调用）和 prompt 体系。

## 5. 反思：多 Agent 架构的本质取舍

本讨论揭示了一个 deeper truth：

> **多 Agent 架构的核心价值是可控性（知识准确、结构完整、可迭代、可审查），而非创意质量。创意视觉设计是一个"整体性任务"——将其分解为多个专业化子任务时，分解本身就会系统性地压制其最珍贵的品质。**

这不是架构的"失败"，而是架构的**取舍**。正如 JSON Schema 为 Agent 间通信提供了确定性但牺牲了自然语言的丰富性，多 Agent 分工为质量提供了可控性但牺牲了设计的自由度。

课程报告中应当诚实地呈现这一发现——它比一个"所有指标都更好"的系统更有学术价值。它展示了：
1. 对系统局限性的清醒认知（而非盲目吹捧自己的架构）
2. 对"为什么分解不如整体"的深层分析（设计委员会效应、IR 信息损失、satisficing 天花板）
3. 基于根因分析的针对性优化方案（而非模糊的"需要更好的 prompt engineering"）
4. 对未来方向的启示（哪些子任务适合分解、哪些不适合——这是 Agentic 系统设计的核心问题）
