## 1. 评估框架

每个 prompt 从以下 5 个维度评估：

| 维度 | 说明 | 权重 |
|------|------|:----:|
| **指令清晰度** | 指令是否明确、无歧义、可被执行 | 25% |
| **约束完整性** | 是否覆盖了所有必要的边界条件和错误场景 | 25% |
| **输出可控性** | 输出格式/内容是否被充分约束，偏差风险低 | 20% |
| **信息密度** | 有用信息占比 vs 冗余/重复内容 | 15% |
| **已知缺陷** | 从实际运行结果中证实的 prompt 层面的问题 | 15% |

每个 prompt 给出：总体评分 (1-10)、优势分析、不足分析、具体优化建议。

## 2. Agent 1 — agent1_system.txt

> **文件**: `src/prompts/agent1_system.txt`
> **角色**: 内容分析器 — 意图分类、NER、RE、知识缺口检测、图表推荐
> **温度**: 0.3

### 2.1 总体评分: 6.5/10

| 维度 | 评分 | 简评 |
|------|:----:|------|
| 指令清晰度 | 8/10 | CoT 步骤清晰，6 步分析链逻辑合理 |
| 约束完整性 | 5/10 | **核心短板**：entity 结构化输出约束不足，导致 NER F1=0.406 |
| 输出可控性 | 5/10 | JSON Schema 完整但模型偏向在 key_points 自由文本中"提及"实体而非填入 entities[] |
| 信息密度 | 7/10 | 步骤 2.5 交叉验证是好设计，但整体篇幅偏长 |
| 已知缺陷 | 6/10 | NER 召回率仅 0.269，RE 召回率仅 0.216（见 results.md §8.3） |

### 2.2 优势分析

1. **六步 CoT 设计合理**：意图分类 → 实体抽取 → 交叉验证 → 关系识别 → 信息完整性 → 受众分析，链条完整且逻辑递进。
2. **步骤 2.5（实体-KeyPoint 交叉验证）**是优秀的设计：在完成 key_points 提取后强制回填实体，理论上应保证实体覆盖。但从实际结果看，模型执行不力。
3. **最低数量要求**（第 2 步）按主题类型差异化：技术类 ≥6 term、历史类 ≥1 人物+4 日期+2 地点、对比类全标注、流程类全步骤节点。这个设计很细致。
4. **知识缺口检测**逻辑明确：信息充分→false，仅给主题→true，有结构化内容→false。
5. **图表类型推荐映射表**覆盖了 6 种意图到 6 种图表的映射关系。
6. **Critical Rules 块**（输出格式前）包含了 7 条硬性规则，包括"key_points 不能为空""entities 必须覆盖 key_points 中的名词"。

### 2.3 不足分析

#### 2.3.1 🔴 根本问题：实体结构化输出极度不足

**数据证据**（results.md §8.3）：

| 样例 | NER F1 | RE F1 | 实体覆盖 |
|------|:------:|:-----:|----------|
| sample1 (LLM原理) | 0.154 | 0.000 | 仅 1/12 实体 (仅 "大语言模型") |
| sample2 (词向量) | 0.308 | 0.000 | 仅 2/11 实体 |
| sample3 (SYSU) | 0.133 | 0.000 | 仅 1/14 实体 |
| sample4 (咖啡链) | 0.800 | 0.909 | 提示词信息充分 → 表现最好 |
| sample5 (数据) | 0.615 | 1.000 | RE 完美, NER 数字格式偏差 |

**根因分析**：

Agent 1 的 `key_points[]` 自由文本中大量提及实体名词（如 sample1 key_points 含 "Transformer"、"自注意力机制"、"预训练"等），但 `entities[]` 结构化数组几乎为空。**模型"知道这些概念"但"没有将其抽取为结构化实体"**。

虽然 prompt 中已有多处约束（步骤 2、步骤 2.5、Critical Rules 块），但存在以下问题：

1. **约束分散在三处**，模型可能只关注到一处而忽略其他。
2. **缺少"示例驱动的强制输出模板"**——当前的 JSON Schema 只定义了字段名和类型，没有给出一个完整的、填满实体的正面示例。LLM 对 "entities: [...]" 的默认理解可能是"列出用户提示词中字面出现的名词"，而非"展开该主题的所有核心概念"。
3. **步骤 2 的"主动补充"指令**（"对于提示词仅给出主题但未列举具体内容的情况……你必须基于你的知识主动补充"）位置在步骤 2 中部，在长篇指令中容易被"稀释"。
4. **步骤 2.5 的交叉验证**虽然是好设计，但它依赖模型自身在步骤 5 之后"回头看"——这在实际推理中容易因注意力衰减而跳过。

#### 2.3.2 🟡 中等问题：关系抽取几乎全部失败

sample1-3 的 RE F1 全部为 0.000。常见问题：

- **target 非实体名**：如 `target="发展历程中的多个时间节点"`（这是一个描述，不是 entity name）
- **source/target 未在 entities 中出现**：违反了 JSON Schema 中的约束
- **关系类型误判或缺失**：对于明显存在层级关系的概念（如 Transformer 包含 Self-Attention），未抽取

prompt 中虽然有"source 和 target 必须是 entities 数组中已存在的实体 name"的要求，但 **没有给出违反此规则的负面示例**，模型可能不理解"怎么算违反"。

#### 2.3.3 🟡 中等问题：意图分类不够具体

sample1 的 prompt 是"解释大语言模型的基本原理"，Agent 1 输出 primary_type="concept_explanation"、secondary_type="科普"。对于技术系统（LLM、Transformer），architecture_diagram 比 concept_explanation 更具体且更适合下游布局规划。

prompt 中缺少 **"具体性优先"的排序规则**：当多个类型同时适用时，架构图 > 概念图 > 流程图 > 时间线 > 对比图。

#### 2.3.4 🟢 轻微问题：数字实体格式不一致

sample5 中 Agent 1 输出了 "10 times more" 作为 number 实体，但 GT 期望的是纯数值 "10"（量词放入 relation.quantifier）。prompt 中对 `number` 类型的定义是"仅数值本身，如 '10'、'2'"——这个定义是正确的，但模型没有严格执行。

### 2.4 优化建议

#### 建议 1.1 🔴 增加"正面示例 + 反例对比"块（预期收益：NER F1 +0.3~0.4）

在 JSON Schema 之前，增加一个**带数据的完整示例**，展示"好的输出"长什么样。示例应该包含 8-10 个正确标注的实体和 5-7 个正确的关系三元组。

```
## ⚠️ 输出质量示例（关键参考）

以下是一个"好"的输出应该达到的密度标准：

提示词："解释大语言模型的基本原理"（仅 10 字）

✅ 正确的 entities 密度（至少 8 个）：
- "大语言模型" (term, primary)
- "Transformer" (term, primary)
- "自注意力机制" (term, primary)
- "前馈神经网络" (term, secondary)
- "词嵌入" (term, secondary)
- "残差连接" (term, secondary)
- "层归一化" (term, secondary)
- "预训练" (term, primary)
- "微调" (term, secondary)
- "RLHF" (term, secondary)

❌ 错误的 entities（仅从提示词字面抽取，太稀疏）：
- "大语言模型" (term, primary)  ← 仅 1 个！不及格！

✅ 正确的 relations（source/target 都是 entity name）：
- {"type": "hierarchy", "source": "大语言模型", "target": "Transformer"}
- {"type": "hierarchy", "source": "Transformer", "target": "自注意力机制"}
- {"type": "sequence", "source": "预训练", "target": "微调"}

❌ 错误的 relations（target 不是 entity name）：
- {"type": "temporal", "source": "中山大学", "target": "发展历程中的多个时间节点"}
  ↑ "发展历程中的多个时间节点" 不在 entities 中！
```

#### 建议 1.2 🔴 将"主动补充实体"提升为独立步骤，使用更强硬的语言

将步骤 2 中的主动补充指令从正文中独立出来，放在 JSON Schema 之前作为**独立检查点**：

```
### ⚠️ 步骤 2 补充检查：实体充分性自查（在输出 JSON 之前必须执行）

在完成所有分析后、输出 JSON 之前，你必须回答以下问题：
1. 我的 entities 数组中有多少个 term 类型的实体？
   → 如果 < 6 且主题是技术概念类，我必须回到步骤 2 重新补充
2. 我的 key_points 中提到的每个专有名词是否都在 entities 中？
   → 如果不是，我必须逐一添加
3. 我的 relations 中每个 source 和 target 是否都能在 entities 中找到？
   → 如果不是，我必须修正（要么添加实体，要么修改关系引用）

如果以上任何一项答案为"否"，不要输出 JSON——先返回相关步骤补充。
```

#### 建议 1.3 🟡 增加意图分类的"具体性排序"规则

```
### 步骤 1 补充规则：具体性优先

当提示词同时匹配多种意图类型时，按以下优先级选择 primary_type：
  architecture_diagram > concept_map > process_diagram > timeline > comparison_chart > concept_explanation

规则解释：
- 如果主题是技术系统（LLM、神经网络、Transformer）→ 优先 architecture_diagram
- 如果主题是抽象概念且不涉及具体架构（如"什么是幸福"）→ 用 concept_explanation
- 如果提示词包含"流程图" → 优先 process_diagram / flowchart
```

#### 建议 1.4 🟢 数字实体格式强化

```
- number：仅数值本身，如 "10"、"2"、"1"、"1924"。**绝对不能包含量词或单位**
  （如"10 times"❌ → 应拆为 number="10" + 量词放在 relation.quantifier="10 times more than"）
```

## 3. Agent 2 — agent2_system.txt

> **文件**: `src/prompts/agent2_system.txt`
> **角色**: 布局规划器 — 布局规划、配色方案、排版、视觉层次
> **温度**: 0.3, max_tokens=32768

### 3.1 总体评分: 7.0/10

| 维度 | 评分 | 简评 |
|------|:----:|------|
| 指令清晰度 | 8/10 | 5 条设计原则 + 4 套配色方案，清晰明确 |
| 约束完整性 | 5/10 | 缺少边界场景处理指导（实体极少/极多、无关系等） |
| 输出可控性 | 6/10 | JSON 字段过多导致 token 消耗大，design_notes 易超长被截断 |
| 信息密度 | 7/10 | content 部分质量高，但 JSON Schema 字段可精简 |
| 已知缺陷 | 7/10 | 曾因 design_notes 过长导致 JSON 截断（已通过 max_tokens=32768 缓解） |

### 3.2 优势分析

1. **比例布局是核心正确决策**：使用百分比而非绝对坐标，从架构上规避了尺寸不匹配问题。
2. **四套配色方案完整**：覆盖科技、自然、学术、数据可视化四大场景，色彩值精确到 hex。
3. **排版方案有 6 级字号体系**，覆盖标题→脚注的完整层次。
4. **设计原则有设计理论支撑**：Visual Hierarchy、Gestalt Proximity、Alignment，并非随意列举。
5. **connections 字段考虑周全**：方向、线型、标签均有定义。

### 3.3 不足分析

#### 3.3.1 🟡 中等问题：JSON Schema 字段过多，token 效率低

elements 数组中要求每个元素填写 `size_hint`（min_width, min_height, aspect_ratio）和 `content_hint`。对于 20+ 元素的架构图，这些字段消耗大量 token 但实际价值有限——Agent 3 有权覆盖这些建议。这是典型的"过度规范"。

**代码证据**（changelog §Phase 2 补充修复）：Agent 2 曾输出 8805 字符 JSON，design_notes 中文长文本导致 max_tokens 截断。

#### 3.3.2 🟡 中等问题：缺少按图表类型的布局策略指导

当前 prompt 只定义了一种通用的布局方法（sections + elements + connections），但不同图表类型需要不同的布局策略：

- **architecture_diagram**：需要明确的分层结构（输入→编码→处理→解码→输出），每层有子元素
- **concept_map**：需要定义"中心概念"和"辐射分支"的拓扑关系
- **timeline**：需要定义时间轴方向（垂直/水平）、节点间距策略（等距 vs 按年份间距）
- **bar_chart**：需要定义 Y 轴基线位置、刻度计算规则

目前这些策略的区分被完全委托给了 Agent 2 的"设计原则"——但 Agent 2 也需要明确的指导。

#### 3.3.3 🟡 中等问题：缺少边界场景处理

- 如果 entities 只有 3 个（极少），如何布局？（可能的方案：放大每个元素，使用中心辐射布局）
- 如果 entities 有 25+ 个（极多），如何降级？（可能的方案：分组折叠、缩小次要元素、使用滚动暗示）
- 如果 relations 为空（无语义关系），如何确定元素的相对位置？
- 如果有 10+ 个 connections，如何避免箭头混乱？

#### 3.3.4 🟢 轻微问题：design_notes 无长度约束

`design_notes` 是自然语言字段，模型可能写出 500+ 字的设计说明。当前通过 max_tokens=32768 和 JSON 重试机制兜底，但治标不治本。应在 prompt 中直接约束：

```
"design_notes": "string (给 Agent 3 的整体设计指导，不超过 150 字。用简短的自然语言描述你想要的整体视觉效果和 2-3 个关键设计决策)"
```

### 3.4 优化建议

#### 建议 2.1 🟡 精简 elements 必填字段

```diff
  "elements": [
    {
      "id": "string (唯一标识)",
      "type": "rect | rounded_rect | circle | ...",
      "label": "string (此元素上显示的文本)",
      "parent_section": "string (所属 section id)",
      "color_key": "string (primary | secondary | ...)",
      "importance": "primary | secondary | tertiary",
-     "size_hint": {
-       "min_width": "number",
-       "min_height": "number",
-       "aspect_ratio": "number | null"
-     },
+     "size_hint": "string (可选，如 'small_icon_24x24' | 'medium_card_200x80' | 'wide_banner_700x60')",
-     "content_hint": "string (Agent 3 应该在此元素中展示什么内容)"
    }
  ],
```

用预定义的尺寸标签替代数值，大幅减少 token 消耗。

#### 建议 2.2 🟡 增加按图表类型的布局策略块

```
## 按图表类型的专项布局策略

### 架构图（architecture_diagram）
- 采用自底向上或自左向右的分层结构
- 每层定义明确的 processing stage 标签
- 残差/跳跃连接从模块侧面出发，沿画布边缘绕行
- 颜色编码：嵌入层=蓝色系、注意力层=紫色系、FFN层=绿色系、输出层=暖色系

### 时间线（timeline）
- 垂直时间线：画布高 1200-1400，节点沿中轴线均匀分布
- 水平时间线：画布宽 1000-1200，时间从左到右
- 关键节点（建校、合并、双一流）使用更大圆点 + accent color
- 非等距时间轴需标注"时间非线性"提示

### 柱状图（bar_chart）
- Y轴基线留 100px 左侧空间用于刻度标签
- 柱子间距 = 柱子宽度的 1.0-1.5 倍
- Y轴最大值向上取整到最近的"好看"数字（如数据最大 20 → Y轴顶 25）
- 图例放在图表右上角或下方

### 流程图（process_diagram）
- 垂直流程节点尺寸统一（推荐 180×55 或 200×60）
- 节点垂直间距 30-50px
- 步骤编号在节点左侧或内部左上角
- 如果步骤 > 7，考虑双列布局
```

#### 建议 2.3 🟢 增加 design_notes 长度硬约束

在 JSON Schema 中：

```
"design_notes": "string (不超过 150 字。只写 2-3 个最关键的设计决策和你想传达的整体氛围)"
```

## 4. Agent 3 — agent3_system.txt

> **文件**: `src/prompts/agent3_system.txt`
> **角色**: SVG 代码生成器 — SVG XML 编写、视觉设计、文本凝练
> **温度**: 0.5, max_tokens=32768

### 4.1 总体评分: 5.5/10

| 维度 | 评分 | 简评 |
|------|:----:|------|
| 指令清晰度 | 6/10 | "SVG 编码铁律"部分强，但设计策略部分过弱 |
| 约束完整性 | 4/10 | **核心短板**：数值推理场景缺少公式化指导，sample5 因此失败 |
| 输出可控性 | 6/10 | 有 markdown 清理 + 坐标自适应权限，但缺少防错机制 |
| 信息密度 | 5/10 | 设计策略每种图表只有 1 行，展开不足 |
| 已知缺陷 | 5/10 | sample5 柱状图比例完全错误（20:2:1 → 4:2:1），sample1 对比度问题 |

### 4.2 优势分析

1. **"SVG 编码铁律"部分质量高**：XML 正确性（7 条）+ 布局约束（5 条）+ 文本规范（6 条）+ 颜色规范（3 条）= 21 条具体可执行的规则。
2. **布局调整权下放是正确的架构决策**：Agent 3 可微调 Agent 2 的坐标建议，自然的容错机制。
3. **"宁可多留白也不要让信息拥挤"**——这个底线约束防止了信息过载。
4. **特殊场景处理覆盖了三种常见情况**：数值关系、多步骤流程、历史时间线。
5. **几何图标组件库**提供了 6 种基础图标的设计模式。

### 4.3 不足分析

#### 4.3.1 🔴 严重问题：数值推理缺少公式化指导

**数据证据**（results.md §5.5）：sample5 v1 中柱体高度为 700:350:175 = 4:2:1，而非正确的 20:2:1（YouTube=10× TikTok, TikTok=2× Kuaishou）。数值标签也全部错误。

**根因**：Agent 3 的 prompt 中有"精确计算比例尺和坐标"的要求，但没有给出**具体的计算公式**。LLM 在不借助外部工具的情况下进行多步数值计算容易出错。

svg_guidelines.txt 中后来补充了"柱状图坐标计算铁律"，但 agent3_system.txt 本身没有显式引用或强化这些公式。

#### 4.3.2 🔴 严重问题：设计策略过于简略

6 种图表类型，每种只有一行设计策略：

```

#### 建议 3.2 🔴 将柱状图计算公式内联到 agent3_system.txt

**第一步：建立数值模型**
1. 找出最小的那个数值，设为基准值 = 1x
2. 根据提示词中的倍数关系，计算所有参与方的倍数：
   - 如 "YouTube 比 TikTok 多 10 倍，TikTok 比 Kuaishou 多 2 倍"
   - → Kuaishou = 1x, TikTok = 2x, YouTube = 20x
3. 检查倍数关系是否自洽：TikTok = 2 × Kuaishou ✓, YouTube = 10 × TikTok = 10 × 2 = 20 ✓

**第二步：计算柱体高度**
1. 确定 Y 轴最大刻度 = 向上取整(max_value × 1.1)
2. 可用绘图高度 = 画布总高 - 坐标轴标签区域 - 上下留白
3. 每个柱子的 height_px = (value / y_max) × 可用绘图高度
4. 每个柱子的 y = baseline_y - height_px

**第三步：标注数值**
1. 每个柱顶标注其倍数（如 "20x"、"2x"、"1x"）
2. 在柱子之间用箭头+文字标注倍数关系（如 "10×" 在 YT→TK 之间）
```

- Agent 3 从 svg_guidelines.txt 中读取配色作为 fallback + 从 Layout IR 获取 Agent 2 的选择

**问题**：如果 Agent 2 选择的配色与 svg_guidelines.txt 中的定义不完全一致（如 hex 值略有差异），Agent 3 会面临冲突。

**建议**：不需要修改代码，但需确保两份文件中的配色值完全一致。目前检查后确认 4 套方案的 hex 值一致 ✅。

### 7.2 排版规范：agent2_system.txt ↔ agent3_system.txt ↔ svg_guidelines.txt

三份文件都定义了排版字号。检查一致性：

| 小节标题 | 20-24px | 20-24px | 20-24px | ✅ |

| 节点标签 | 14-16px | 14px | 14-16px | ⚠️ 轻微差异 |

agent3 将"标签"统一定为 14px，而 agent2 和 svg_guidelines 是 14-16px 范围。建议统一为 "14-16px" 以给 Agent 3 留有调整空间。

### 7.4 信息传递的断层：Agent 1 的 NER 低质量如何影响下游

这是当前架构中最重要的跨 Agent 一致性问题。Agent 1 的 entities[] 不完整（F1=0.406），但 Agent 2 的 `build_user_prompt` 仅遍历 entities 中 `importance=primary/secondary` 的实体来构建布局提示（layout_planner.py:79-81）。这意味着：

- 如果 Agent 1 漏了 entity，Agent 2 就看不到它
- 漏掉的实体不会出现在 Layout IR 的 elements 中
- Agent 3 也就不会为它生成 SVG 元素

但幸运的是，Agent 2 同时也会看到 `key_points` 和 `knowledge_supplement`，这些自由文本中通常包含了完整的实体信息。所以 Agent 2 可以通过理解文本来弥补实体数组的缺失——**但这正是我们希望 IR 结构避免的"隐性信息传递"**。

**结论**：修复 Agent 1 的 NER 质量是最高优先级，因为它直接影响整个流水线的信息传递完整性。
